Docker basics Q&A

Posted on Thu, Dec 10, 2020 docker

Docker basics

  • What is a docker client (docker CLI)?

    It is a way of communicating with the docker architecture and docker components. It allows us to send commands to the docker server (docker deamon). It doesn't do anything related to dockers except of sending the messages further to be executed.

  • What is a docker deamon (docker server)?

    It is a tool responsible for creating images, running containers etc. - a heart of the docker ecosystem.

  • What is a docker hub?

    It is a repository of container images, including containers from other developers.

  • How does docker run hello-world work?

    It goes with a few simple steps:

    1. first, the command docker run hello-world is parsed using docker client.
    2. docker client sends the information to the docker server to get the image.
    3. docker server checks whether there is a local image available in so called 'image cache'.
    4. if so, it runs that image, if not, it tells us that there is no local copy of that image
    5. it next reaches out to the docker hub, downloads that image, caches it locally and runs it.
💡

(Unix) Operating systems use kernel to communicate between running processes and hardware. The process makes 'system call' that is processed by kernel. Namespacing is segmenting a part of the hard disk to host a specific environment, e.g. with a specific python version. It enables two programs running in different environments to run together on the same system. Above is one of the uses of namespacing. Control groups are used to limit amount of resources dedicated to a single process. These two are used together to create a 'separated' part of the hardware that is used by selected processes.

  • What is a container?

    It is a process or family of processes that have a separated group of resources specifically assigned to it. It is also a 'running image' - it is an instance of the image, concept similar to object of a class.

  • What is an image?

    It is a snapshot of the file system used to run a certain program/programs with a running command that is executed to run the container. When running the container based on an image, the kernel is isolating a specific part of hard drive for the container and puts the resources there. Next, the running command is executed.

💡

Docker is running on windows/mac via linux virtual machine host, that's why it's possible to use the idea of namespacing.

  • How can you override the docker run command?

    By using the docker run <image-name> <command> syntax. For instance docker run busybox ls will list the files in the snapshot of busybox docker image.

💡

Notice that the command is executed inside of the container environment. It means that only the commands that are included in the filesystem snapshot of a docker image will be executed; otherwise it will print an error message. For example, the hello-world docker image has only one program inside, hence 'echo' or 'ls' functions will fail to work when overriding the run command.

  • How can you list docker containers currently running?

    By using docker pscommand. To see all of the containers ever created, execute docker ps --all

  • How does docker run work?

    It runs docker create + docker start. Using docker create prints the id of the created docker (it only maps the snapshot to the hard drive). Next, docker start <id> runs the container. docker start -a <id> also prints the output from the run command and container logs(if there are any).

💡

We can use docker start to run again the exited container. However, we cannot override the run command that was used when first creating the container. Hence, after using docker run or docker create we can only re-run that container without changing the command.

💡

Use docker run -d <id> to run the docker in the background and be able to still use the terminal to run some commands. Same goes for docker-compose up -d and other commands that are blocking the terminal.

  • How can we delete all of the created containers?

    By using docker system prune command. Be wary, it also will remove them irreversibly with all cached images also. To remove single container, run docker container rm <id> .

  • How can we get information about container?

    By using docker logs <id> command.

  • What is the difference between docker stop and docker kill ?

    The first one asks container 'nicely' to stop. It gives it 10 second to finish process before sending the sigkill. The other stops the container immediately.

  • How can we run a single command (another process) inside of running container?

    It is done by using docker exec -it <id> <command> syntax. The -i flag allows the user to interact with the docker (opens stdin channel), while -t flag conveys the output in a 'terminal-like' manner.

💡

docker exec -it <id> sh opens up the shell terminal mode inside of the container (if it's available). This allows us to type multiple commands just like in unix shell. docker run -it <image-name> sh is overriding the basic run command with the shell init command. So it doesn't run the primary run command but allows us to issue some command in terminal right away. Quit the shell with ctrl+d.

Creating a docker image

  • Describe docker building using a Dockerfile.

    Standard Dockerfile flow consists of:

    1. Specifying a base image that we will build on (command FROM).
    2. Running additional commands on top of the base image (command RUN).
    3. Adding a run command for container startup (command CMD).

    The docker building runs command by command from Dockerfile and creates temporary docker containers, runs commands inside and creates snapshots. It also caches the in-between docker images so they can be used for another build.

    To build a docker image run docker build . command.

    To tag the docker (so we can use arbitrary name instead of the id) we can use building with

    docker build -t maciejdomagala/<image-name>:latest . command to tag our docker with the latest version and call it by the name.

    💡

    To define a Dockerfile that will be used for a guild we should use the -f flag. E.g docker build -f Dockerfile.dev

  • Describe building using existing image (not Dockerfile)

    We can create docker images from running containers. We are doing that by using the command docker commit -c 'CMD ["<run-command>"]' <existing-container-id>.

    This creates an image with existing container as base and arbitrary run-command specified above. Even tho it is possible to use that, it is advised to use Dockerfile type of image creating for consistency purposes.

  • How do we copy local files to the container?

    By using COPY <src> <dest> syntax inside Dockerfile. If we are running Dockerfile in the project folder COPY ./ ./ would do.

  • How can we choose the place where files are headed when copying?

    By pointing to specific directory when using COPY or by using WORKDIR <path> command. The latter is setting up the directory from which the commands inside the container will be run. Hence COPY ./ ./ would actually copy the files from ./ locally to <path>/.

Working with Docker Compose

  • What is Docker Compose and why do we use it?

    Docker compose is a docker tool used in multi-container applications. It is making cumbersome and repetitive using of docker CLI automatic. It is used in a form of YAML file (named docker-compose.yaml). The file contains configuration of services necessary for the whole app that is based on several containers.

    We use docker compose as a more suitable way of executing the docker cli commands by moving them to the .yaml file.

    Docker manages single container, docker-compose - multi-containers.

  • Explain the docker-compose.yaml from below.

  1. We state the version of compose tool. More on that here: https://docs.docker.com/compose/compose-file/compose-versioning/
  2. Services would be the containers to run in parallel that we need for the app. One of them is redis-server run from the 'redis' image, second is the node-app, which is built from the working directory Dockerfile.
  3. For the node-app container we also state the ports that will be mapped from local port to the container port.
💡

Use docker-compose up and docker-compose down to start and stop the container network defined in docker-compose.yaml. Use the -d flag to run it in the background.

  • What are the restart policies for docker-compose?
    1. default 'no' - never try to restart the network.
    2. 'always' - always restart the containers.
    3. 'on failure' - only restart when we get status code other than 0.
    4. 'unless-stopped' - try to restart unless the developer stopped it.
💡

Use docker-compose ps from the directory with docker-compose.yaml file to get information about the containers status (containers linked to docker-compose.yaml file).
